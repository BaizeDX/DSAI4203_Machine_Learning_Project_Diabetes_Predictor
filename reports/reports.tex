\begin{abstract}
This project presents a machine learning approach for predicting diabetes using the Kaggle Diabetes Prediction Challenge dataset. The dataset contains 700,000 training samples and 300,000 test samples with 26 clinical and demographic features. We develop a systematic modeling pipeline that progresses from baseline models to an optimized XGBoost model with proper cross-validation. Key contributions include: (1) comprehensive exploratory data analysis revealing class imbalance and feature correlations; (2) progressive model development from simple baselines to advanced gradient boosting; (3) critical methodological improvements including 5-fold stratified cross-validation, fold-wise categorical encoding, and out-of-fold evaluation; (4) feature engineering based on clinical domain knowledge; and (5) model interpretation using feature importance analysis. The final XGBoost model achieves a mean cross-validation AUC of 0.7256 with low variance (0.0008), demonstrating stable generalization performance. The complete pipeline is implemented in a modular Python package with full reproducibility.
\end{abstract}

\section{Introduction}

\subsection{Background}
Diabetes is a chronic metabolic disorder affecting millions worldwide. Early detection and risk assessment are crucial for timely intervention. Machine learning models can assist healthcare professionals by identifying high-risk individuals based on clinical measurements and demographic information.

\subsection{Problem Statement}
Given a dataset of clinical measurements and demographic information, the task is to predict whether an individual has been diagnosed with diabetes. The evaluation metric is ROC-AUC, which measures the model's ability to distinguish between positive and negative cases across all classification thresholds.

\subsection{Dataset Description}
The dataset is synthetically generated from a deep learning model trained on the Diabetes Health Indicators Dataset. It consists of:
\begin{itemize}
    \item Training set: 700,000 samples, 26 features
    \item Test set: 300,000 samples, 25 features (target excluded)
    \item Target variable: \texttt{diagnosed\_diabetes} (binary: 0 = no diabetes, 1 = diabetes)
\end{itemize}

\section{Exploratory Data Analysis}

\subsection{Target Distribution}
The target variable shows imbalance: 62.3\% of samples are diabetic (class 1) and 37.7\% are non-diabetic (class 0). This imbalance ratio of 1.65 indicates that class 1 is the majority.

\subsection{Feature Correlations}
Correlation analysis reveals that \texttt{family\_history\_diabetes} has the strongest positive correlation with the target (0.211), followed by \texttt{age} (0.161) and \texttt{systolic\_bp} (0.107). Physical activity shows a negative correlation (-0.170), consistent with medical knowledge.

\subsection{Categorical Features}
Six categorical features are present: gender, ethnicity, education level, income level, smoking status, and employment status. No missing values are present in the dataset.

\subsection{Key Observations}
\begin{itemize}
    \item No missing values, simplifying preprocessing
    \item Class imbalance requires careful evaluation strategy
    \item Strong predictive signals from family history and age
    \item Outliers exist in physical activity and sleep hours features
\end{itemize}

\section{Methodology}

\subsection{Baseline Models}

\subsubsection{Dummy Classifier}
A dummy classifier predicting the majority class (diabetic) establishes a lower bound for performance. This achieves an AUC of 0.5000, representing random classification.

\subsubsection{Decision Tree}
A decision tree with max depth 5 serves as an interpretable baseline. It achieves an OOF AUC of 0.6825, with feature importance dominated by \texttt{family\_history\_diabetes} (44.8\%), \texttt{physical\_activity} (29.9\%), and \texttt{age} (21.8\%).

\subsection{Model Selection Considerations}

Before committing to XGBoost, we evaluated several candidate models. Decision tree served as our initial baseline, but its shallow depth limited its ability to capture complex feature interactions. We also experimented with K-Nearest Neighbors (KNN) and Multi-Layer Perceptron (MLP). KNN struggled with the high-dimensional feature space and large dataset size, while MLP required extensive hyperparameter tuning and was computationally expensive. XGBoost emerged as the best choice due to its ability to handle tabular data effectively, built-in regularization to prevent overfitting, and efficient handling of categorical features. The gradient boosting framework naturally captures feature interactions and provides feature importance for interpretability.

\subsection{XGBoost Model Development}

\subsubsection{Initial Model}
The initial XGBoost model with default parameters (100 trees, learning rate 0.1) achieves AUC 0.7192, significantly outperforming the decision tree baseline.

\subsubsection{Model Capacity Tuning}
Increasing tree count to 300 improves AUC to 0.7255, while further increasing to 500 yields 0.7264. Learning curves show convergence beyond 500 trees, indicating diminishing returns.

\subsubsection{Feature Engineering}
Based on clinical domain knowledge, five engineered features were initially added. However, we made a critical mistake in Experiment 07: we replaced the original features with interaction features (e.g., using \texttt{age\_family\_history} while dropping \texttt{age} and \texttt{family\_history\_diabetes}). This caused information loss—when family history was zero, the interaction term became zero regardless of age. The corrected approach retains all original features while adding engineered features as supplements.

The final feature set includes:
\begin{itemize}
    \item \texttt{age\_family\_history}: interaction between age and family history
    \item \texttt{age\_bmi}: interaction between age and body mass index
    \item \texttt{cardio\_risk\_score}: composite cardiovascular risk score
    \item \texttt{cholesterol\_ratio}: total cholesterol divided by HDL
    \item \texttt{non\_hdl\_cholesterol}: total minus HDL cholesterol
\end{itemize}

\subsection{Critical Methodological Improvements}

\subsubsection{Why We Moved to Cross-Validation}
Early experiments (Experiments 03-10) all used a single fixed validation split with \texttt{random\_state=42}. While convenient, this approach risks overfitting to that specific split. The model's hyperparameters were tuned repeatedly on the same validation set, potentially learning patterns specific to that data partition rather than general patterns. This is a classic pitfall in machine learning—tuning on a fixed holdout set can lead to optimistic performance estimates that don't generalize.

\subsubsection{From Fixed Holdout to Cross-Validation}
To address this, the final pipeline employs 5-fold stratified cross-validation. The data is randomly partitioned into 5 folds while preserving class distribution. Each fold serves as the validation set exactly once, with the remaining 4 folds used for training. This ensures every sample is validated exactly once, providing a more realistic estimate of generalization performance.

\subsubsection{Fold-Wise Categorical Encoding}
To prevent data leakage, categorical encoding is performed within each fold:
\begin{lstlisting}[language=Python]
def _encode_categorical_fold(X_train_raw, X_valid_raw, categorical_cols):
    for col in categorical_cols:
        le = LabelEncoder()
        le.fit(X_train_raw[col])      # Fit only on fold-train
        X_train_encoded[col] = le.transform(X_train_raw[col])
        X_valid_encoded[col] = le.transform(X_valid_raw[col])
\end{lstlisting}

\subsubsection{Out-of-Fold Evaluation}
OOF (Out-of-Fold) predictions collect validation predictions from all folds, providing an unbiased estimate of generalization performance. Each sample's OOF prediction comes from a model that never saw it during training.

\subsubsection{Early Stopping}
Training stops when validation AUC fails to improve for 100 consecutive rounds, automatically determining optimal tree count and preventing overfitting.

\subsection{Final Model Architecture}
The final XGBoost model uses optimized hyperparameters:
\begin{itemize}
    \item \texttt{n\_estimators}: 2000 (early stopping active)
    \item \texttt{learning\_rate}: 0.03
    \item \texttt{max\_depth}: 4
    \item \texttt{subsample}: 0.8
    \item \texttt{colsample\_bytree}: 0.8
    \item \texttt{gamma}: 1 (split regularization)
    \item \texttt{reg\_alpha}: 0.1, \texttt{reg\_lambda}: 1 (L1/L2 regularization)
\end{itemize}

\subsubsection{Advantages of Modular Design}

The module architecture provides several benefits over the notebook-only approach:

\begin{enumerate}
    \item \textbf{Separation of Concerns}: Each file handles a specific responsibility—\texttt{config.py} manages paths and parameters, \texttt{features.py} handles feature engineering, \texttt{models.py} contains training logic. This makes the code easier to understand and maintain.
    
    \item \textbf{Reusability}: Functions can be imported and reused across different experiments without code duplication. The same feature engineering pipeline can be applied to both training and test data consistently.
    
    \item \textbf{Testability}: Individual components can be tested in isolation. For example, we can verify that \texttt{add\_engineered\_features()} produces the expected columns without running the full training pipeline.
    
    \item \textbf{Command-Line Interface}: The module can be executed from the terminal, making it suitable for automated workflows and batch experiments.
    
    \item \textbf{Version Control}: Changes are tracked more cleanly in modular code than in monolithic notebooks. Git diffs are more meaningful when modifications are confined to specific files.
    
    \item \textbf{Documentation}: Each function has a clear docstring, making the code self-documenting and easier for collaborators to understand.
\end{enumerate}

The final module is organized as:

\begin{lstlisting}[language=bash]

\end{lstlisting}

This executes the complete pipeline: data loading, feature engineering, baseline evaluation, XGBoost CV training, and submission generation.

\section{Discussion}

\subsection{Key Findings}

\begin{enumerate}
    \item \textbf{Cross-validation is essential}: The transition from fixed holdout to 5-fold CV revealed consistent performance, eliminating overfitting concerns. The low standard deviation (0.0008) across folds confirms that our performance estimates are reliable.
    
    \item \textbf{Feature engineering adds value}: Domain-informed features like \texttt{cardio\_risk\_score} entered the top 5 importance rankings, validating the approach. However, we learned that replacing original features with interactions (as in Experiment 07) causes information loss—interactions should supplement, not replace.
    
    \item \textbf{Interaction effects matter}: The \texttt{age\_family\_history} interaction contributes nearly as much as the original family history feature, indicating non-linear relationships that simple linear models would miss.
    
    \item \textbf{Model stability}: Low CV variance (0.0008) confirms the model's generalization capability, and the close alignment between mean CV AUC (0.7256) and OOF AUC (0.7256) indicates robust evaluation.
    
    \item \textbf{Modular design improves reproducibility}: The transition from notebook to modular Python package produced identical results while enabling easier maintenance, testing, and automation.
\end{enumerate}

\subsection{Lessons Learned}

\subsubsection{What Worked}
\begin{itemize}
    \item Stratified cross-validation for reliable evaluation
    \item Fold-wise categorical encoding preventing data leakage
    \item Early stopping for automatic model complexity control
    \item Domain-informed feature engineering
    \item Retaining original features while adding interactions
    \item Modular code architecture for reproducibility
\end{itemize}

\subsubsection{What Could Be Improved}
\begin{itemize}
    \item Further feature engineering (e.g., polynomial interactions)
    \item Hyperparameter search over larger space
    \item Ensemble methods combining XGBoost with LightGBM
    \item SHAP analysis for enhanced interpretability
\end{itemize}

\section{Conclusion}

This project developed a systematic machine learning pipeline for diabetes prediction, progressing from simple baselines to an optimized XGBoost model with proper cross-validation. The final model achieves a mean CV AUC of 0.7256 with low variance (0.0008), demonstrating stable generalization. Key methodological contributions include fold-wise categorical encoding, out-of-fold evaluation, and early stopping for automatic model selection.

The pipeline was successfully refactored from a notebook into a modular Python package (\texttt{src/}), with verification confirming identical results. This modular design improves reproducibility, maintainability, and reusability—essential qualities for production-ready machine learning systems.

The project highlights the importance of rigorous evaluation protocols in machine learning and demonstrates how domain knowledge can be effectively integrated through feature engineering. While the Kaggle test score (0.6956) is slightly lower than CV performance due to data distribution shifts, the model's consistent cross-validation results and stable performance across folds provide confidence in its generalization capability.
