## 2. What were the main difficulties encountered by the whole team? and by your responsible task?

### Team difficulties

* **Synthetic data distribution shift** – The dataset was generated by a deep learning model, not from real clinical records. This caused a noticeable gap between our internal cross‑validation mean AUC (0.72585) and the final Kaggle public score (0.69626).
* **Class imbalance** – 62.33% diabetic (positive class) vs. 37.67% non‑diabetic (negative class). We relied on ROC‑AUC (which is insensitive to imbalance) because that is the competition evaluation metric.
* **Methodological pitfalls** – Early experiments reused the same fixed holdout split for tuning, risking overfitting. We later corrected this by moving to 7‑fold cross‑validation, which provided a mean AUC of 0.72585 with a fold‑level standard deviation of 0.00129.

### My responsible task difficulties

* **Feature engineering mistake** – In an intermediate experiment, I initially replaced original features with interaction features (e.g., using `age_family_history` while dropping `age` and `family_history_diabetes`). This caused information loss because when `family_history_diabetes = 0`, the interaction term becomes zero regardless of age. The correction was to **retain all original features** and **add** engineered features as supplements. This lesson is documented in the report: "strong original predictors must be retained in full, with engineered variables added selectively on top."
* **Global vs. fold‑wise categorical encoding** – At first, I applied `LabelEncoder` to the entire training set before cross‑validation, which leaks validation‑fold information into the encoder. Fixing this required moving the encoding **inside each fold**, fitting the encoder only on the fold‑training data. This was the most subtle but critical methodological fix.
* **Code duplication** – Feature engineering was accidentally applied twice (once in `train_xgb_cv.py` and again in `models.py`). I cleaned this up to ensure a single source of truth.
* **Missing value handling order** – Using `astype(str).fillna('Unknown')` would turn `NaN` into the string `"nan"` before filling. The correct order is `fillna('Unknown').astype(str)`. (Note: the final dataset had no missing values, but this was addressed for pipeline robustness.)

## 3. What have you learned from this competition?

### Technical lessons

* **Cross‑validation is not optional** – A single holdout set can give optimistic results. The comparison of 5‑fold, 7‑fold, and 10‑fold CV (Notebook 13) revealed that 7‑fold offers the best balance of performance (0.72585), stability (std = 0.00129), and computational cost (505 seconds).
* **Fold‑wise preprocessing matters** – Any operation that learns from data (encoding, scaling, imputation) must be fitted **inside each fold** to avoid leakage.
* **Feature engineering is about supplementing, not replacing** – Interaction features should be added, not substituted for original features. The top features in the final model (`family_history_diabetes` at 42.5%, `age_family_history` at 29.7%, `age_bmi` at 6.8%, `cardio_risk_score` at 5.0%) demonstrate that both original predictors and carefully designed interaction features contribute meaningfully.
* **Early stopping is a simple but powerful regulariser** – With n_estimators=2000 and early stopping, the model automatically finds the optimal number of trees (mean best iteration ~1996 across folds) and prevents overfitting.
* **Modular code > monolithic notebooks** – Refactoring into a package (`src/`) improved reproducibility, made debugging easier, and forced me to think clearly about data flow.

### Project management lessons

* **Keep a detailed experiment log** – Recording every parameter change, CV score, and observation helped me track what worked and what didn't. This was essential for writing the final report.
* **Fix methodological errors early** – The mistake of global encoding and feature replacement cost time to debug. Running a quick sanity check (e.g., checking that the number of features stays consistent) would have caught some of them earlier.
* **Separate training code from visualisation** – The final pipeline produces raw results (`logs/`), and a separate script (`report_figures.py`) generates all report charts. This made it easy to regenerate figures without re‑training.

### Final takeaway

This competition taught me that **a clean, correct pipeline is more valuable than chasing tiny AUC gains**. The final model's 7‑fold CV mean AUC (0.72585) is only marginally better than early holdout experiments, but the evaluation is now **trustworthy** because of proper cross‑validation, fold‑wise encoding, and systematic fold‑count comparison. I also learned that **documenting mistakes** is as important as documenting successes – it shows a genuine learning process.
